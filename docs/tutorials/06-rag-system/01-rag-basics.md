# 01. RAG 基础：为什么需要 RAG

> RAG（Retrieval-Augmented Generation）可能是最近两年 LLM 应用开发里最重要的范式。这篇从"为什么"讲起，再说"什么时候用"。

---

## 1. LLM 的知识限制

先说一个问题：你让 GPT-4o 给你查一下今天沪深 300 的收盘价，它能查到吗？

答案是不能。原因很简单——**LLM 的知识是静态的**。

即使是最新的模型，它的训练数据也有一个截止日期。它学到的所有知识，都是在某个时间点之前从互联网上抓取的内容。这意味着：

- 它不知道刚刚发生的新闻
- 它不知道你公司内部的 API 文档
- 它不知道你昨天刚写完的代码
- 它不知道某个私有的业务数据

这就像一个人参加完高考之后就再也没学过任何新东西——他的知识很丰富，但全是"历史知识"。

### 幻觉问题

比知识过时更严重的是**幻觉**。当 LLM 不知道答案时，它不会诚实地说"不知道"，而是会**编造**一个看起来合理的答案。

```
用户：2024 年诺贝尔物理学奖得主是谁？
模型：约翰·霍普菲尔德和杰弗里·辛顿
```

这个答案看起来像那么回事，但它可能是错的（取决于时间点）。更可怕的是，模型说这句话时的语气和它说"1+1=2 一样笃定"。

> RAG 的核心价值：**让 LLM 在回答时，能引用真实、新鲜、可靠的外部数据，而不是凭空编造。**

## 2. 什么是 RAG

RAG 就是一个简单的流程：**先检索，再生成**。

```
用户问题
    │
    ▼
┌──────────────┐     ┌──────────────┐
│   检索阶段    │ ──► │   生成阶段    │
│   从知识库    │     │   LLM 结合   │
│   找到相关    │     │   检索结果   │
│   片段        │     │   生成回答   │
└──────────────┘     └──────────────┘
    │                       │
    ▼                       ▼
  知识库                  最终答案
```

实际上，RAG 的流程可以拆成四步：

1. **用户提问**：用户输入一个问题
2. **检索**：从知识库中找到最相关的内容片段
3. **拼接**：把检索到的内容拼到 LLM 的上下文里（作为参考材料）
4. **生成**：LLM 基于"原始问题 + 参考材料"生成最终答案

这个过程对用户是完全透明的——用户只看到一个问题和一个答案，背后的检索逻辑对他不可见。

![配图：RAG 四步流程 - 小黑在流水线上从书架找书交给 LLM 回答](/assets/06-rag-02-rag-flow.png)

## 3. 什么时候需要 RAG

不是所有场景都需要 RAG。做技术最忌"拿着锤子看什么都像钉子"。

### 推荐使用 RAG 的场景

| 场景 | 例子 | 为什么需要 RAG |
|------|------|---------------|
| **实时数据** | 股价、天气、新闻、比赛比分 | LLM 知识过期，无法回答 |
| **私有知识** | 公司内部文档、产品手册、客户数据 | 这些数据从未出现在训练集中 |
| **高准确率要求** | 医疗诊断、法律咨询、金融报告 | 需要引用原文，不能靠回忆 |
| **频繁更新的知识** | 产品文档（每周更新）、政策法规 | 重新训练模型成本太高 |
| **多语言/跨领域** | 中文查英文资料、技术查行业规范 | 检索可以跨越语言和领域障碍 |

### 不需要 RAG 的场景

| 场景 | 例子 | 为什么不需要 |
|------|------|-------------|
| **通用常识** | "Python 的列表怎么用" | LLM 训练数据里已经有了 |
| **创意生成** | "给我写一首诗" | 不需要外部事实支撑 |
| **简单翻译** | "把这个翻译成英文" | LLM 本身就能做好 |
| **纯逻辑推理** | "解这个数学题" | 不需要外部知识 |

### 一个判断标准

> **如果你能接受 LLM 用自己的"记忆"来回答这个问题，那就不需要 RAG。如果你希望答案基于特定的、真实的、最新的事实，那就需要 RAG。**

## 4. 一个简单的 RAG 流程示例

```javascript
/**
 * 最简 RAG 流程
 * 三步：检索 → 拼接 → 生成
 */
async function simpleRAG(userQuestion, knowledgeBase) {
  // 1. 检索：从知识库中找到相关片段
  const relevantDocs = await retrieveRelevant(
    knowledgeBase, 
    userQuestion, 
    topK = 3
  );

  // 2. 拼接：把检索结果和问题拼在一起
  const context = relevantDocs
    .map((doc, i) => `[参考 ${i + 1}]：${doc.text}`)
    .join("\n\n");

  const prompt = `
你是一个知识助手。请基于以下参考材料回答问题。
如果参考材料不足以回答，请明确说"材料中未提及"。

参考材料：
${context}

问题：${userQuestion}
`;

  // 3. 生成：LLM + 参考材料 = 最终答案
  const response = await callLLM([
    { role: "user", content: prompt }
  ]);

  return response;
}
```

这个例子里，`retrieveRelevant` 函数就是 RAG 的核心——它决定了你能找到多好的参考材料。下一篇文章会深入讲这个函数的各种实现方式。

## 6. 面试高频问法

**Q1：为什么 LLM 需要 RAG？仅仅微调不行吗？**

微调和 RAG 解决的**不是同一个问题**。

- 微调让模型**学新的行为或风格**（比如让它学会用特定格式回答问题）
- RAG 让模型**获取新的事实信息**（比如最新的产品文档）

微调不能解决时效性问题——就算你每周微调一次，最新的一批数据在微调之后还是过时了。而且微调成本高、周期长，不适合频繁更新的知识。实践中，**RAG 和微调是互补关系**：RAG 负责实时事实，微调负责行为和风格。

**Q2：RAG 的流程是什么？**

四步：用户提问 → 从知识库检索相关片段 → 拼接到 LLM 上下文 → LLM 生成最终答案。关键在于"检索"这一步的质量，它直接决定了 RAG 的效果。

**Q3：什么时候需要 RAG，什么时候不需要？**

需要：实时数据、私有知识、高准确率要求、频繁更新的知识、跨领域查询。
不需要：通用常识、创意生成、简单翻译、纯逻辑推理。

判断标准：看你是否需要答案"基于特定的事实"还是"基于模型的记忆"就可以了。

**Q4：RAG 和直接给 LLM 一个大上下文的区别是什么？**

直接给大上下文（比如把整本产品手册塞到 system prompt 里）的问题是：

- **上下文窗口有限**：GPT-4 的上下文可能装不下你的全部知识库
- **成本高**：token 越多，API 费用越高
- **注意力稀释**：模型在大量文本中很难精准定位需要的信息（"大海捞针"问题）
- **无法扩展**：知识库从 100 条增长到 10000 条，你不能让上下文也跟着涨 100 倍

RAG 通过检索只给模型"最相关的几段话"，既节省 token，又提高准确性。
