 # 什么是 Harness
 
 ## 一个直觉
 
 想象你在跟一个 AI 模型聊天。你说一句，它回一句。这很直接。
 
 但如果你想让它做一件复杂的事——比如"帮我写一个计算器网页，然后打开浏览器测试"——光靠一次对话是不够的。它需要：
 
 1. 先写 HTML
 2. 写 CSS
 3. 写 JS
 4. 打开浏览器看效果
 5. 发现按钮位置不对，回去改 CSS
 6. 再刷新浏览器
 
 这个"写代码→执行→看结果→再写"的循环，就是 **Agent 循环**。而**支撑这个循环能跑起来的那层中间件，就叫 Harness**。
 
 ## 不是什么黑话
 
 "Harness" 在英文里的本意是"马具"——把马的力量传导到车上那个连接件。用在软件里也是类似的含义：
 
 ```
   LLM 模型（动力源）
       ↓
   [ Harness ]  ← 传导层
       ↓
   工具 / 文件系统 / 终端（车轮）
 ```
 
 没有 Harness，模型再强也只能吐文字。有了 Harness，模型能调终端命令、读写文件、调用 API、观察结果、自己决定下一步做什么。
 
 ## 不同框架里的叫法
 
 | 框架 / 产品 | 叫法 | 本质 |
 |---|---|---|
 | Codex CLI | agent_v2.js 的核心循环 | 一个 while 循环 + tool routing |
 | LangChain | AgentExecutor | Agent + Tool 的编排器 |
 | LangGraph | StateGraph 运行时 | 基于图的状态机驱动 |
 | OpenAI | Assistants API / Run | 服务端托管的执行环境 |
 | Anthropic | Tool Use + 循环 | 客户端自己实现的 loop |
 | AutoGPT | AI Agent 循环 | 最朴素的 while-true 循环 |
 
 本质上都在做同一件事：**把 LLM 的"聊天接口"封装成一个"干活接口"。**
 
 ## 一个更精确的定义
 
 Harness 是一个运行层，它：
 
 - **驱动循环**：不断地"让模型思考 → 解析模型输出 → 执行工具 → 把结果塞回上下文 → 让模型再思考"
 - **管理上下文**：控制发给模型的 tokens，防止上下文爆炸
 - **调度工具**：把模型声明的工具调用，翻译成真实的系统操作
 - **处理错误**：工具超时了、结果太大了、模型跑偏了——这些都不是模型该操心的事
 - **暴露接口**：给上层（Plan Mode、用户交互）一个统一的执行入口
 
 所以回到开头那个问题——什么是 Harness？
 
 一句话：**让 LLM 从"聊天玩具"变成"干活工具"的那层胶水。**
